## What Worked

app/auth/callback/page.tsx

'use client'

import { useEffect, useRef } from "react"
import FHIR from 'fhirclient'

export default function CallbackPage() {
    const isReadyCalled = useRef(false)

    useEffect(() => {
        FHIR.oauth2.ready().then(client => {
            // This stops the second 'double-fire' in Next.js Dev mode
            if (isReadyCalled.current) return; 
            isReadyCalled.current = true;

            client.patient.read().then(data => {
                console.log("Patient Data:", data)
            })

            client.patient.request("MedicationRequest").then(data => console.log("Medications:", data))
            client.patient.request("AllergyIntolerance").then(data => console.log("Allergies:", data))
            client.patient.request("Observation?category=laboratory").then(data => console.log("Labs:", data))
        });
    }, []);

    return <div>Completing authorization...</div>
}

Building on yesterday's working `client.patient.read()` call, adding `client.patient.request()` for each clinical resource type was straightforward. Each call returns a FHIR Bundle — the same `searchset` structure we reviewed in the introduction — scoped to the patient in context.

## What broke / what confused me

**403 Forbidden on resource requests**

For an extended stretch, every `MedicationRequest`, `AllergyIntolerance`, and `Observation` request returned a `403 Forbidden`. The code looked correct, the patient had data, and permissions appeared configured. The root cause turned out to be in the Epic app registration form, not the code.

Because our scopes use the SMART v2 `.rs` syntax — where `r` is read and `s` is search — Epic requires both `Read` and `Search` to be explicitly registered as separate API permissions. Registering only `MedicationRequest.Read (R4)` isn't sufficient; `MedicationRequest.Search (R4)` must also be selected. This applies to every resource type. The 403 wasn't an auth failure — it was a missing permission that Epic enforces at the resource operation level.

One additional note: after updating API permissions in the Epic developer portal, changes can take up to an hour to propagate. Plan accordingly — it's a good window for reading Epic's FHIR documentation or reviewing the resources your app will be querying.

**"Another process already logged in" error**

After resolving the 403 issues, the launch sequence was failing at login with:
```
server_error: You cannot authenticate while there is another process already logged in on the same browser.

The culprit was React's Strict Mode, which is enabled by default in Next.js development builds. Strict Mode intentionally invokes useEffect twice to surface side effect bugs — which meant the OAuth launch was firing twice, completing once, and then failing on the second invocation with a session conflict.
The fix was introducing a useRef flag (isReadyCalled) that short-circuits the second invocation. useRef persists across renders without triggering a re-render, making it the right tool for this pattern. This is a development-only issue — Strict Mode's double-invocation behavior doesn't occur in production builds — but it needs to be handled cleanly during the build phase.

## What I learned
Epic treats Read and Search as distinct permissions. This is one of those implementation details that isn't obvious from the FHIR spec alone. A GET /MedicationRequest?patient=123 is a search operation, not a read — even though you're reading data. If your SMART v2 scope includes .rs but your app registration only grants Read, Epic will return a 403 on every patient-scoped query. Register both for every resource type you intend to access.

React Strict Mode will break OAuth flows if not handled. The useRef guard pattern is the correct solution here and worth keeping in the codebase as a documented decision rather than a silent fix.
