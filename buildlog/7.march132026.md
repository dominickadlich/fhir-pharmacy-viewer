# Build Log — March 13 2026

## What I Attempted
Refactored the renal dosing review pipeline from a single-prompt call to a full agentic loop with four tools: get_active_antibiotics, get_renal_labs, calculate_creatinine_clearance, and lookup_dosing_protocol. Rather than embedding all clinical context upfront in a monolithic prompt, the model now requests only the data it needs, when it needs it.

## What Worked
The refactor introduced three distinct layers.

app/lob/tools.ts - tool definitions

import Anthropic from "@anthropic-ai/sdk";

export const tools: Anthropic.Tool[] = [
    {
        name: "get_active_antibiotics",
        description: "Returns the patient's active antibiotic orders that require renal dose adjustment",
        input_schema: {
            type: "object",
            properties: {},
            required: []
        }
    },
    // ... additional tools
];

executeToolCall() — tool execution logic

The tool definitions themselves are lightweight — names, descriptions, and input schemas. The clinical logic lives in executeToolCall(), a switch statement that maps each tool name to its corresponding function and data source. The separation is intentional: the model decides which tool to call and when; the application controls what that tool actually does.

app/api/renal-review/route.ts — agentic loop

The POST handler now runs a while loop that continues until the model's stop_reason is "end_turn" rather than "tool_use". On each iteration where the model requests a tool, the results are appended to the conversation and the loop continues:

if (response.stop_reason === "tool_use") {
    const toolResults: Anthropic.ToolResultBlockParam[] = response.content
        .filter((block): block is Anthropic.ToolUseBlock => block.type === "tool_use")
        .map(block => ({
            type: "tool_result" as const,
            tool_use_id: block.id,
            content: executeToolCall(
                block.name,
                block.input as Record<string, unknown>,
                patient,
                medications,
                observations
            )
        }));

    messages.push({ role: "user", content: toolResults });
}

## What broke / what confused me
Silent prop mismatch — medication vs. medications

A typo in the callback file passed medication (singular) where the tool execution function expected medications (plural). TypeScript didn't catch it because the prop shape was still technically valid — it just resolved to undefined. The model returned a response, but without medication data. No error, no stack trace. The output was the only signal something was wrong. Lesson: when a model returns a coherent but clinically hollow response, check your data pipeline before the prompt.

Explicit return types on tool definitions

TypeScript inferred the return type of the tools array as a generic object array rather than Anthropic.Tool[]. This caused a type error downstream in route.ts when the array was passed to the SDK. Explicitly annotating const tools: Anthropic.Tool[] resolved it. When working with SDK types, don't rely on inference — annotate explicitly.

setState is asynchronous — don't use freshly set state in the same closure

After calling setRenallyDosedAbx(), the subsequent fetch body referenced renallyDosedAbx from state — which was still the empty initial value. React state updates are asynchronous and won't reflect in the same execution context where they were set. The fix is to store the value in a local variable and pass that directly into the fetch body. Never reference state that was just set in the same closure.

## What I learned
Tool definitions are contracts, not implementations.

The structure of a tool file is simpler than it looks. get_active_antibiotics and get_renal_labs need only a name, a description, and an input schema. The model uses the description to decide when to call the tool — precision there matters. The logic itself lives in executeToolCall(), which is just a switch statement routing tool names to functions. The architecture is clean precisely because the two responsibilities are separated.

The agentic loop is more intuitive than it sounds.

The while loop pattern reads naturally once you understand what's happening: the model runs, requests data, receives it, runs again, and repeats until it has enough context to produce a final response. It's a conversation with a decision-maker who asks clarifying questions before committing to a recommendation. In clinical terms, it maps to a pharmacist pulling the chart before calling the provider — gathering context before acting.

Role framing in the system prompt is an architectural decision.

The model flagged the hardcoded 70kg weight assumption as unverified. Not because LLMs are generically cautious in healthcare contexts — but because the system prompt established a clinical pharmacist role with explicit accountability for dosing recommendations. When you frame the model as a pharmacist operating under an institutional protocol, it inherits the epistemic standards of that role. It doesn't guess at patient weight; it asks. That behavior isn't a model quirk — it's the system prompt working as intended. Clinical judgment gets encoded at the role definition layer, not at the instruction layer.
