## What broke / what confused me
Server-to-client data passing in Next.js:

My first attempt placed the API call to Claude in a separate async server component. Next.js doesn't allow server component data to be passed directly to client components as props — the boundary is strict. Moving the fetch call into the useEffect hook inside the callback page resolved this cleanly, since the entire component is already client-side.

Sending the wrong data shape to the model:

The initial prompt was constructed with data.patient — the raw FHIR Patient resource — rather than the parsed representation. The model received a deeply nested FHIR object with references, extensions, and identifiers rather than clean demographic fields. Instead of hallucinating a response, the model correctly stated it lacked sufficient context to generate a recommendation. That behavior caught the bug. Swapping data.patient for parsePatient(data.patient) gave the model the structured input it needed and the recommendations followed immediately.This is worth noting: a well-prompted model failing gracefully on bad input is a feature, not a bug. It's the LLM equivalent of a typed function rejecting a malformed argument.

## What I learned
Prompt structure is an engineering decision, not a stylistic one.

The pattern that produced reliable output was:

Role — establish the model's clinical context
Policy — embed the institutional renal dosing protocol
Patient context — structured demographic and lab data
Instruction — the specific clinical question
Output format — explicit constraints on how the response should be structured

Each layer narrows the model's response space. Omitting any one of them produced noticeably degraded output.

Output format specification is not optional.

The model's default behavior was to return Markdown — headers, bold text, bullet points. Readable in a document, not in a plain UI div. The tempting fix was Tailwind's prose class to render Markdown in the browser. The correct fix was a single prompt constraint: "Do not use Markdown formatting. Write in plain prose only." One instruction, no additional dependencies.

The LLM is downstream of your data pipeline. The quality of the model's recommendation is a direct function of what you send it. A raw FHIR Bundle is noise. A parsed, typed patient object with labeled lab values and a relevant institutional policy is signal. Garbage in, garbage out applies here exactly as it does everywhere else in software.
