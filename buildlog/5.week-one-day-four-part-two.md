The clinical logic driving the conflict banner lives in a shared utility:
app/lib/fhir/utils.tsx

export function findDrugAllergyConflicts(
    medications: ParsedMedication[] | undefined,
    allergies: ParsedAllergy[] | undefined
): {
    drug: string;
    allergy: string
}[] {
    const conflicts: { drug: string; allergy: string }[] = []

    medications?.forEach(med => {
        allergies?.forEach(allergy => {
            if (med.name.toLocaleLowerCase().includes(allergy.code.toLocaleLowerCase())) {
                conflicts.push({ drug: med.name, allergy: allergy.code });
            }
        });
    });

    return conflicts;
}

This performs a case-insensitive string match between each medication name and each documented allergen. It's a naive implementation by design — the goal at this stage is to validate the UI pattern, not to build a production-grade interaction engine.

## What broke / what confused me
React props are objects, not arrays. This caused multiple TypeErrors when I attempted to call .map() directly on a prop. A component receives its data as an object — if you're passing an array, it lives on a key inside that object. Destructure it correctly or access it explicitly before mapping.

Ternary operators belong in useEffect, not in the return statement. Placing conditional logic inside the JSX return means it re-evaluates on every render. Moving it into the useEffect call ensures it runs once when the data is available and the result is stored in state — which is what drives the render cleanly.

Finding a real test patient with a drug-allergy conflict is harder than it sounds. Epic's sandbox patients are synthetic, and true allergy-medication conflicts are rare in the dataset by design. This blocked UI testing longer than it should have.

## What I learned
It's okay to use dummy data for UI development. Waiting for the perfect test patient is a form of procrastination. If you need a drug-allergy conflict to test the banner, manufacture one — hardcode a medication name that matches an allergen, confirm the component renders correctly, then delete it before launch. Testing infrastructure is not production data.

Build your own components before reaching for an LLM. The TypeError debugging, the prop destructuring confusion, the ternary placement — those are the moments where understanding actually forms. An LLM will give you working code immediately, but working code you don't understand is technical debt you'll pay later. Struggle with it first.

The string-matching approach to drug-allergy detection is a starting point, not a solution. A production pharmacy decision support tool would use RxNorm relationships and allergen class mappings — not substring matching on display names. "amoxicillin-clavulanate" won't match an allergy documented as "penicillin" with this logic, even though the clinical risk is real. That's the next layer of clinical sophistication this project will need.
